Service Portal widgets are ServiceNow-flavored Angular 1, which translates to "dark arts squared". It can be a challenge for a developer of any level of experience to map online knowledge of Angular widgets to how they are built vs the Service Portal, and one of the biggest wrinkles is understanding how the client controller cooperates with the server script.

If you're new to widgets, the following may be of some help. For best results, create a temporary widget in a PDI and copy the examples below to follow along.

## Data context and life cycle

A widget exists to present data so that data needs to be sourced. Before a widget is rendered, it depends on a _data context_ that is generated by first running the server script before being propagated into the controller. You'll know this as the global variable `data` in the server script and as `c.data` in the controller script template.

```js
// Server
/*global data*/
(function serverScript() {
  var ga = new GlideAggregate("incident");
  ga.addQuery("active", true);
  ga.addAggregate("COUNT");
  ga.query();
  data.activeIncidentCount = ga.next() ? parseInt(ga.getAggregate("COUNT")) : 0;
})();
```

```js
// Controller
api.controller = function ($scope) {
  var c = this;
  // Map fields onto the scope for binding to the template
  $scope.activeIncidentCount = c.data.activeIncidentCount;
};
```

```html
<!-- Template-->
<div>There are {{activeIncidentCount}} active incidents</div>
```

This completes a full run of the widget update cycle: The server script runs before the initial presentation, the client script massages the data context and exposes it to the template, and the template defines bindings that finally present the data to the user.

In many cases, this may be all that you need, but if your client controller needs to have a conversation with the server the relationship between these scripts becomes less clear: The client can request the server script to be run _again_, which will then update the `c.data` object on the controller with new information. For brevity, here's a nightmarish example of how this can work:

```js
// Server
/*global data*/
(function serverScript() {
  data.timestamp = Date.now();
})();
```

```js
// Controller
api.controller = function ($scope, $timeout) {
  var c = this;
  (function updateTimestamp() {
    $scope.timestamp = c.data.timestamp;
    // Run the server script again and report back when finished
    c.server.update().then(function () {
      $timeout(updateTimestamp, 1000); // Repeat in a second
    });
  })();
};
```

```html
<!-- Template-->
<div>Current timestamp: {{timestamp}}</div>
```

This will show a timer that increments every second, and it does so in a hugely inefficient do-not-do-this manner by polling the server for timestamps by repeatedly executing the server script with `c.server.update()`.

> Note: While it's possible to skip the controller in its entirety by having the template bind directly onto `{{c.data.activeIncidentCount}}` I prefer to explicitly map values from `c.data` to `$scope` in the controller. This helps keep in mind that the data context is a communications channel between the server and the client and reinforces knowledge of its contents. It also helps the template's readability.

> Server updates and data bindings directly against `c.data` allow you to create some "magical" solutions that may feel cool at the moment but become difficult to follow as your data context grows and the fields you want to be presented become intermixed with fields you just need for controller logic.

## Update vs Get

We have two ways to request a server update from the client controller; `server.update` and `server.get`. The difference between the two depends on what you are trying to achieve: Do you simply want to run any cached queries again, or do you want to pass user inputs back to the server for processing? Both `update` and `get` can achieve the same outcomes in terms of processing inputs and receiving responses, but the context differs slightly.

Common to both is that they will prompt a re-run of the server script but with a new global variable: `input`. The existence of the `input` object is what we use to discern whether the server script is being run _implicitly_ before the widget is first rendered or because the controller _explicitly_ requested it.

Let's look at the two side by side:

```js
// Server
/*global data, input*/
(function serverScript() {
  if (input) {
    /* We have input, so we are being called explicitly by the client controller
        Process the input fields and (optionally) write results back to the data context */
    data.text = input.text.toUpperCase();
  }
})();
```

The big difference here is in the client controller. If we use `get` we declare exactly what `input` is going to be on the server. If we use `update` the entirety of `c.data` is passed back as the `input` object. Additionally, if we call `get` the server script does _not_ automatically update `c.data`, whereas `update` will.

```js
// Controller
api.controller = function ($scope, $timeout) {
  var c = this;

  // server.get formally declares an input object
  $scope.capitalizeWithGet = function () {
    if ($scope.text) {
      // Do we have text to capitalize?
      // create an object of input values to pass
      var input = {
        text: $scope.text,
      };
      // Pass it to the server
      c.server.get(input).then(function (result) {
        /* Process the new data retrieved from the server. 
                    Note that c.data has not changed. */
        $scope.text = result.data.text;
      });
    }
  };

  /* 
    	server.update implicitly declares an input object by passing 
    	the current client controller data context (c.data)
    */
  $scope.capitalizeWithUpdate = function () {
    if ($scope.text) {
      // Do we have text to capitalize?
      // Put the text we want to capitalize on c.data
      c.data.text = $scope.text;
      // Poke for a server update
      c.server.update().then(function () {
        // c.data is now updated, write it back to the scope
        $scope.text = c.data.text;
      });
    }
  };
};
```

```html
<!-- Template-->
<form>
  <input ng-model="text" />
  <button ng-click="capitalizeWithGet()">Capitalize with get</button>
  <button ng-click="capitalizeWithUpdate()">Capitalize with update</button>
</form>
```

Both approaches can be useful.

If your template binds directly onto values in `c.data` and you never really need to pass input back to the server a call to `server.update` may be what you are after.

If you fly by my approach where you manually map the data context from `c.data` onto `$scope`, a `get` is almost always preferable since it removes ambiguity about what it is that you pass as input and demonstrates intent in what you want to allow the server to process and what you expect in return.

## Formalizing a communications channel

This model of communication can create server scripts full of unpleasant-looking conditionals. You must first check for `input`, then check if the field(s) you care about are present, and then produce some output.

```js
// Server
/*global data, input*/
(function serverScript() {
  if (input) {
    if ("foo" in input) {
      // handle foo
    } else if ("bar" in input) {
      // handle bar
    }
    // ..etc
  }
})();
```

You may also wind up passing flags to identify the type of update ("submitting", "refreshing"), or come up with a messaging protocol where you do a switch by message type, something like this:

```js
// Server
/*global data, input*/
(function serverScript() {
  function handleFormSubmit(inData) {
    return {
      formValid: true,
    };
  }
  function refreshData() {
    return {
      // some fields here
    };
  }

  if (input) {
    if ("type" in input) {
      data = (function () {
        switch (input.type) {
          case "form_submitted":
            return handleFormSubmit(input);
          case "refresh_clicked":
            return refreshData();
          default:
            throw "Unknown message type " + input.type;
        }
      })();
    }
  } else {
    data = refreshData();
  }
})();
```

For better or worse, ServiceNow leaves us some legroom for innovation here. I'm sure there are different solutions for how to handle this stuff for every team that ever created a widget. I concluded that what I wanted was a messaging protocol where the controller calls named functions on the server receives their return values as a promise, and codifies the step where the server script performs initial setup.

## The Server/Client message bus

My solution has two dependencies; An angular service provider that passes messages to the server script and a serverside script-include that makes it easier to create handlers for those messages.

```js
// Server
/*global data, input, ServerMessageBus*/
(function serverScript() {
  function handleInput() {
    // Define a dictionary of message handlers
    var handlers = {
      capitalize: function (input) {
        return input.toUpperCase();
      },
    };
    ServerMessageBus.update(input, data, handlers);
  }

  function performInitialSetup() {
    // Place static/cached values on data here
  }

  if (input) {
    handleInput();
  } else {
    performInitialSetup();
  }
})();
```

```js
// Controller
api.controller = function (ClientMessageBus, $scope, $timeout) {
  var c = this;
  var messages = ClientMessageBus.create(c);
  //TODO: Perform initial map of static/cached c.data values to $scope here

  $scope.capitalize = function () {
    if ($scope.text) {
      messages.sendMessage("capitalize", $scope.text).then(function (result) {
        $scope.text = result;
      });
    }
  };
};
```

At this point, you can probably guess that we are using `server.get` behind the scenes here. The key is that we wrap each call to `get` in a metaphor where we are expecting a specific thing to happen on the server that will produce some expected kind of result.

In my opinion, this makes it easier to follow the relationship between the template, controller and server. It also makes it simpler to explain what's going on in a given widget if it needs server interactions: "The click handler calls capitalize on the controller, the controller calls capitalize on the server".

### ServerMessageBus

```js
var ServerMessageBus = (function () {
  /*
    var exampleMessageHandlers = {
        ping: function (_) {
            return 'pong';
        },
        mul: function(inData){
            return inData.reduce(function(product, operand){
                return product * operand;
            }, inData.pop());
        }
    };
    */

  function processMessage(m, messageHandlers) {
    var handler = messageHandlers[m.name];
    if (handler == null) throw "No handler for message type " + m.name;
    return handler(m.value);
  }

  function update(input, data, messageHandlers) {
    if (input) {
      /* It's safe to just use data.value for every message 
            here since we expect calls from server.get */
      data.value = processMessage(input, messageHandlers);
    }
  }

  return {
    update: update,
  };
})();
```

### ClientMessageBus

```js
function ClientMessageBus() {
  // For use in client controllers vs ServerMessageBus script include
  function create(controller) {
    function sendMessage(name, data) {
      return controller.server
        .get({
          name: name,
          value: data,
        })
        .then(function (result) {
          return result.data.value;
        });
    }
    return {
      sendMessage: sendMessage,
    };
  }
  return {
    create: create,
  };
}
```

## Summary

From what I can tell there's no real "correct" way to do any of this stuff. However, ServiceNow is a platform that makes developers context switch a _lot_, and this is most pronounced when mentally modeling the client/server relationship.

To be productive while avoiding the organic growth of technical debt we need programming models and ways of thinking that minimize the context switching and create common concepts.

I feel pretty good now about what's going on between the server and the client when building moderately complex widgets, and I hope this has been helpful.
